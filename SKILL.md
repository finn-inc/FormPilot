# フォーム自動送信スキル

SES企業のお問い合わせフォームを自動巡回し、定型データを入力・送信するスキル。
Playwright MCP のブラウザ操作を使ってフォームの検出・入力・送信・スプレッドシート更新を自動ループする。

---

## 呼び出し方

```
/form-submit                            → 未送信企業を最大 config.options.maxCompanies 件まで連続処理
/form-submit --company サイバーフリークス  → 特定企業のみ処理
```

引数: $ARGUMENTS
- `--company {企業名}` が指定された場合、その企業のみを処理対象とする
- 引数なしの場合、未送信企業を先頭から順に処理する

---

## 設定ファイル

### `~/.claude/form-submit/config.json`

実行前に必ず Read ツールで読み込む。ネスト構造の詳細は `references/config-schema.md` を参照。

主要キー：
- `config.spreadsheet.url` : 対象スプレッドシートのURL
- `config.spreadsheet.columns` : 各列の列名（companyName / submitted / formUrl）
- `config.spreadsheet.sheet` : シート名
- `config.options.maxCompanies` : 1バッチの最大処理件数（デフォルト 150）
- `config.options.skipOnError` : エラー時にスキップして続行するか（デフォルト `true`）
- `config.options.confirmBeforeSubmit` : 送信前に確認を挟むか（デフォルト `false`）
- `config.options.screenshotAfterSubmit` : 送信後にスクリーンショットを保存するか（デフォルト `true`）
- `config.commonData` : フォームに入力する定型データ（キーバリュー）

### `~/.claude/form-submit/field-log.json`

未知フィールドへの対応を永続記録するファイル。実行前に読み込み、ループ終了後に保存する。
スキーマ詳細は `references/config-schema.md` を参照。ファイルが存在しない場合は `{ "entries": [] }` として扱う。

## 全体フロー（バッチ処理）

### ステップ 1: 設定・ログ読み込み

Read ツールで以下を読み込む：
1. `~/.claude/form-submit/config.json` → `spreadsheet.url`、`options`、`commonData` を取得
2. `~/.claude/form-submit/field-log.json` → 未知フィールドの過去回答を取得
3. `~/.claude/form-submit/progress.json` → 前回の最終処理行を取得

4. Playwright MCP のプロファイル設定を確認する:
   - Playwright MCP の設定ファイル（`~/.claude/plugins/marketplaces/claude-plugins-official/external_plugins/playwright/.mcp.json`）を Read ツールで読み込む
   - `--user-data-dir` 引数が設定されているか確認する
   - 設定されていない場合: ユーザーに「Chromeプロファイルが未設定です。reCAPTCHA通過率が大幅に下がりますが続行しますか？」と確認する
   - 設定されている場合: プロファイルパスが存在するか確認し、問題なければ続行する

**progress.json が存在する場合:**
ユーザーに選択肢を提示する：
```
前回は行{lastRow}（{lastCompany}）まで処理済みです。
[1] 続きから処理する（行{lastRow + 1}から）
[2] 開始行を指定する
```

**progress.json が存在しない場合:**
ユーザーに開始行を直接聞く：
```
progress.json が見つかりません。何行目から開始しますか？
```

### ステップ 2: Googleログイン状態の確認

スプレッドシートに遷移する前に、Chromeプロファイルが正しくGoogleにログインしているか確認する。

1. `browser_navigate` で `https://accounts.google.com` に遷移する
2. `browser_snapshot` でページ状態を確認する
3. **ログイン済みの場合**（アカウント情報やプロフィール画像が表示）: そのままステップ3へ進む
4. **アカウント選択画面の場合**（複数のGoogleアカウントが一覧表示）: ユーザーに「複数のGoogleアカウントが検出されました。使用するアカウントをブラウザ画面で選択してください。完了したら教えてください。」と通知し、選択完了まで待機する。完了後、`browser_snapshot` で再確認してからステップ3へ進む
5. **未ログインの場合**（ログインフォームやメールアドレス入力欄が表示）:
   - ユーザーに通知する: 「Googleにログインされていません。reCAPTCHA通過率が低下します。ブラウザ画面でGoogleにログインしてください。完了したら教えてください。」
   - ユーザーが「完了」と回答するまで待機する
   - ログイン完了後、`browser_snapshot` で再確認してからステップ3へ進む

### ステップ 3: スプレッドシートから企業リスト取得

1. `browser_navigate` で `config.spreadsheet.url` に遷移する
2. `browser_snapshot` でスプレッドシートの構造を確認する
3. `config.spreadsheet.columns` の定義に従い、A列（企業名）・B列（送信済みチェック）・C列（フォームURL）を読み取る
4. B列が未チェック（空欄またはFALSE）の行を全件抽出し、企業リストを作成する
5. `--company` オプション指定時は該当企業名の行のみを対象とする

### ステップ 4: 企業ループ開始

未送信企業を先頭から順番に処理する。バッチ処理の効率を最大化するため、個々のフォーム入力・送信にはユーザー確認を挟まない（未知フィールドへの質問のみが中断ポイント）。

進捗は都度表示してユーザーが状況を把握できるようにする：

```
[3/150] 株式会社XXX を処理中...
```

成功した企業名はメモリ内リストに追記し、ループ終了後にスプレッドシートをまとめて更新する（1社ごとにスプレッドシートへ戻るとブラウザ往復コストが高いため）。

## 1社あたりのステップ

### ステップ 5: フォームURLに遷移

`browser_navigate` でC列のURLに遷移する。

### ステップ 6: フォーム検出

1. `browser_snapshot` でページ構造を取得する
2. `<input>` / `<textarea>` / `<select>` 等のフォームフィールドが存在するか確認する
3. フィールドが見つかった場合: そのままステップ7へ進む
4. フィールドが見つからない場合: フォーム探索ロジック（後述）を実行する

### ステップ 7: フィールド照合

`browser_snapshot` で取得したフィールド一覧と `commonData` のキーを意味的にマッチングする。
Claude 自身がフォームのラベル名と `commonData` キーの意味的対応を判断する。

意味的マッチングの例：
- 「御社名」「貴社名」→ `commonData["会社名"]`
- 「お名前」「担当者名」→ `commonData["氏名"]`
- 「E-mail」「メール」→ `commonData["メールアドレス"]`
- 「TEL」「電話」→ `commonData["電話番号"]`
- 「ご用件」「お問い合わせ種別」→ `commonData["お問い合わせ種別"]`

### ステップ 8: 未知フィールド対応

各フィールドについて以下を判断する：

**マッチ済みフィールド:** `commonData` の対応値を自動使用する

**未知フィールド（`commonData` にも意味的に近いキーがない場合）:**
1. `field-log.json` の `entries` を検索し、同じまたは類似した `fieldLabel` のエントリを探す
2. ログに記録あり → そのエントリの `value` を使用し、`count` を +1、`companies` に当該企業名を追加する
3. ログに記録なし → `AskUserQuestion` ツールでユーザーに質問する：

```
[未知フィールド] 「{フィールドラベル}」への入力値を教えてください。
（このフォームでは選択肢: {選択肢リスト} があります）
```

ユーザーの回答を `field-log.json` に新規エントリとして追加する（`count: 1`）。

### ステップ 9: フォーム入力

フィールドの種類に応じて入力する：
- テキスト入力欄: `browser_fill_form` または `browser_type`
- チェックボックス・ラジオボタン: `browser_click`
- セレクトボックス: `browser_select_option`

バッチ処理の効率を最大化するため、個々の入力にユーザー確認は挟まない。

### ステップ 10: 送信前スクリーンショット

フォーム入力が完了したら、送信ボタンを押す前に `browser_take_screenshot` でフォーム全体のスクリーンショットを撮影する。入力内容の証跡として後から確認できるようにするため。

保存先: `~/.claude/form-submit/screenshots/{企業名}_before.png`

企業名にファイル名として使えない文字（`/` 等）が含まれる場合は `_` に置換する。

### ステップ 11: 送信・確認画面対応

日本のフォームは「確認画面→送信」の2段階が一般的なため、完了まで最大3回のクリックを試みる：

1. 送信ボタン（「送信」「Submit」「確認する」「次へ」等）を `browser_click` でクリック
2. `browser_snapshot` でページ状態を確認する
3. **確認画面の場合**（「入力内容の確認」「以下の内容で送信します」等が表示）:
   「送信する」「上記内容で送信」「確定する」等のボタンを `browser_click` でクリック
4. `browser_snapshot` を再取得し、完了画面になるまで繰り返す（最大3回）

### ステップ 12: 結果確認

1. `config.options.screenshotAfterSubmit` が `true` の場合は `browser_take_screenshot` でスクリーンショットを撮影する
   保存先: `~/.claude/form-submit/screenshots/{企業名}_after.png`（企業名にファイル名として使えない文字が含まれる場合は `_` に置換する）
2. `browser_snapshot` で送信後のページ状態を確認する
3. 以下のいずれかで成功を判断する：
   - 「送信完了」「ありがとうございました」「お問い合わせを受け付けました」等のメッセージが表示されている
   - URLが完了ページ（`/thanks`、`/complete`、`/done` 等）に遷移している
4. 成功と判断できない場合はステップ 13（リトライ＆エラー時の処理）に進む

### ステップ 13: リトライ＆エラー時の処理

**リトライルール（最大3回）:**

エラーが発生した場合、同じ企業に対して最大3回までリトライする。

1. 1回目の試行でエラー → ページをリロードして2回目を試行
2. 2回目もエラー → フォームURLに再遷移して3回目を試行
3. 3回目もエラー → その企業をパス（スキップ）する

リトライ中に成功した場合は、成功企業リストに追加し、`progress.json` の `results` に `"success"` + リトライ回数を記録する。

**3回失敗した場合:**
- エラー内容（企業名・エラー種別・詳細・試行回数）をメモリ内のエラーリストに追加する
- スプレッドシートのB列は更新しない
- 次の企業の処理に進む（ステップ 5 に戻る）

`config.options.skipOnError: false` の場合:
- 3回リトライしても失敗した時点でバッチ処理を停止する

---

## ループ終了後の処理

### ステップ 14: スプレッドシートを一括更新

個々の送信後に毎回スプレッドシートへ戻るとブラウザ往復コストが高くなるため、全社処理完了後にまとめてB列を更新する：

1. `browser_navigate` で `config.spreadsheet.url` に遷移する
2. `browser_snapshot` でスプレッドシートの状態を確認する
3. メモリ内の成功企業リストを順番に処理し、A列を検索して各企業の行を特定する
4. B列のチェックボックスセルを `browser_click` でクリックしてONにする
5. 全件完了後に `browser_snapshot` で変更が反映されたことを確認する

### ステップ 15: フィールドログ保存

処理中に更新した `field-log.json` の内容を Write ツールで保存する。
パス: `~/.claude/form-submit/field-log.json`

### ステップ 16: config昇格提案

`field-log.json` の `entries` の中で `count >= 3` のエントリを抽出し、以下の形式でユーザーに提案する：

```
以下のフィールドが3回以上出現しています。config.json の commonData に追加しますか？

- 「部署名」→ 値: 営業部（5回出現）
- 「ご担当者名カナ」→ 値: ヤマダ タロウ（3回出現）

[Y] すべて追加  [n] スキップ  または追加するフィールドを番号で指定してください
```

ユーザーが承認した場合は `config.json` の `commonData` に該当キーバリューを追加して Write ツールで保存する。

### ステップ 17: 進捗記録

セッション終了時、スプレッドシートの最後に処理した行番号と各企業の結果を `~/.claude/form-submit/progress.json` に記録する。
次回セッション開始時にこのファイルを読み込み、前回の続きから処理を再開できるようにする。

```json
{
  "lastRow": 42,
  "lastCompany": "株式会社XXX",
  "updatedAt": "2026-03-04T15:30:00+09:00",
  "results": [
    { "row": 5, "company": "株式会社AAA", "status": "success", "retries": 0 },
    { "row": 8, "company": "株式会社BBB", "status": "success", "retries": 2, "note": "2回目でフォーム再遷移後に成功" },
    { "row": 12, "company": "株式会社CCC", "status": "failed", "retries": 3, "error": "CAPTCHA検出" },
    { "row": 15, "company": "株式会社DDD", "status": "failed", "retries": 3, "error": "送信エラー: 必須項目未入力" }
  ]
}
```

- `status: "success"` — 送信成功（リトライ中に成功した場合も含む）
- `status: "failed"` — 3回リトライしても失敗した企業。`error` にエラー内容を記録

### ステップ 18: 最終レポート表示

`progress.json` の `results` を集計し、以下の形式で出力する：

```
=== 送信結果 ===
処理範囲: 行5〜行42
成功: 18社（うちリトライ後成功: 3社）
失敗（3回リトライ後パス）: 2社
スキップ（フォーム未検出）: 3社
---
成功詳細:
- 行5  株式会社AAA: 1回目で成功
- 行8  株式会社BBB: 3回目で成功（2回目でフォーム再遷移後に成功）
- 行10 株式会社EEE: 2回目で成功
  ...
---
失敗詳細:
- 行12 株式会社CCC: CAPTCHA検出（3回試行）
- 行15 株式会社DDD: 送信エラー: 必須項目未入力（3回試行）
---
config昇格候補（3回以上出現）:
- 「部署名」→ 営業部（5回出現）→ config.jsonに追加済み ／ スキップ済み
---
進捗: 行42まで完了 → progress.json に保存済み
```

## フォーム探索ロジック

URLがフォームページでない場合（ステップ6でフォームが見つからない場合）に実行する。

1. `browser_snapshot` の現在のページから以下のキーワードを含むリンクを検索する：
   「お問い合わせ」「Contact」「パートナー」「協業」「提携」「SES」「採用」「ビジネス」
2. 最も関連性が高いリンクを選択し、`browser_click` で遷移する
3. 遷移先で `browser_snapshot` を再取得し、フォームフィールドの有無を確認する
4. フォームが見つかればステップ7へ進む
5. 1階層目でも見つからない場合: 同様のキーワードで2階層目まで探索する
6. 2階層探索してもフォームが見つからない場合: スキップしてエラーリストに「フォーム未検出」として記録する

## エラーハンドリング

### CAPTCHA対応

`browser_snapshot` の結果に reCAPTCHA / hCaptcha /「ロボットではありません」等のCAPTCHA要素が含まれる場合、以下の順序で対応する：

**reCAPTCHA v2（チェックボックス型）の場合:**
1. `browser_snapshot` で reCAPTCHA の iframe 内にあるチェックボックス（「私はロボットではありません」）の `ref` を特定する
2. `browser_click` でチェックボックスをクリックする
3. 2〜3秒待機後、`browser_snapshot` でページ状態を確認する
4. **チェックだけで通過した場合**（チェックマークが表示され、画像チャレンジが出ない）: そのままフォーム入力・送信を続行する
5. **画像チャレンジが表示された場合**（「信号機を選択してください」等の画像選択UI）: 突破不可能なためスキップし、エラーリストに「CAPTCHA検出（画像チャレンジ）」として記録する

**reCAPTCHA v3（非表示型）の場合:**
- 通常はフォーム送信時に自動評価されるため、そのまま送信を試みる
- 送信後にCAPTCHAエラーが返された場合: ページをリロードして再試行する（リトライルールに従う）

**hCaptcha / その他のCAPTCHAの場合:**
→ スキップしてエラーリストに「CAPTCHA検出」として記録する

### タイムアウト

`browser_navigate` または `browser_snapshot` が想定時間内に完了しない場合:
→ スキップしてエラーリストに「タイムアウト」として記録する

### ログイン要求

`browser_snapshot` の結果にログインフォーム（`type="password"` フィールドや「ログイン」「サインイン」ボタン）が含まれる場合:
→ スキップしてエラーリストに「ログイン要求」として記録する

### 送信エラー

送信後のページに「エラー」「入力内容をご確認」「必須項目」「送信に失敗」等のメッセージが含まれる場合、またはフォームが再表示されている場合:
→ スキップしてエラーリストに「送信エラー: {エラーメッセージ内容}」として記録する

### フォーム未検出

→ フォーム探索ロジック（上記）のステップ6に従う

## 前提条件（reCAPTCHA 対策）

- Playwright MCP は既存の Chrome プロファイル（Google ログイン済み）を使用する設定になっている
- **スキル実行前に Google Chrome を閉じること**（プロファイルロック競合を回避するため）
- Chrome を閉じ忘れた場合はエラーになるので、ユーザーに閉じるよう案内する

## 人間的な操作パターン

bot 検出を回避するため、フォーム入力時に以下を実施:
- 各フィールド入力間に 500ms〜1500ms のランダム遅延（`browser_run_code` で `await page.waitForTimeout()`）
- 送信ボタンクリック前に 1〜2秒の待機

---

## 注意事項

- スプレッドシートの操作は Google Sheets API ではなく、Playwright MCP 経由のブラウザ操作で行う
- 未知フィールドへの質問には `AskUserQuestion` ツールを使用する
- 処理中は成功企業リストをメモリに保持し、ループ終了後にスプレッドシートへ一括反映する（毎回戻ると非効率）
