# フォーム自動送信スキル

SES企業のお問い合わせフォームを自動巡回し、定型データを入力・送信するスキル。
Playwright MCP のブラウザ操作を使ってフォームの検出・入力・送信・スプレッドシート更新を自動ループする。

---

## 呼び出し方

```
/form-submit                            → 未送信企業を最大 config.options.maxCompanies 件まで連続処理
/form-submit --company サイバーフリークス  → 特定企業のみ処理
```

---

## 設定ファイル

### `~/.claude/form-submit/config.json`

実行前に必ず Read ツールで読み込む。ネスト構造の詳細は `references/config-schema.md` を参照。

主要キー：
- `config.spreadsheet.url` : 対象スプレッドシートのURL
- `config.spreadsheet.columns` : 各列の列名（companyName / submitted / formUrl）
- `config.spreadsheet.sheet` : シート名
- `config.options.maxCompanies` : 1バッチの最大処理件数（デフォルト 150）
- `config.options.skipOnError` : エラー時にスキップして続行するか（デフォルト `true`）
- `config.options.confirmBeforeSubmit` : 送信前に確認を挟むか（デフォルト `false`）
- `config.options.screenshotAfterSubmit` : 送信後にスクリーンショットを保存するか（デフォルト `true`）
- `config.commonData` : フォームに入力する定型データ（キーバリュー）

### `~/.claude/form-submit/field-log.json`

未知フィールドへの対応を永続記録するファイル。実行前に読み込み、ループ終了後に保存する。
スキーマ詳細は `references/config-schema.md` を参照。ファイルが存在しない場合は `{ "entries": [] }` として扱う。

## 全体フロー（バッチ処理）

### ステップ 1: 設定・ログ読み込み

Read ツールで以下を読み込む：
1. `~/.claude/form-submit/config.json` → `spreadsheet.url`、`options`、`commonData` を取得
2. `~/.claude/form-submit/field-log.json` → 未知フィールドの過去回答を取得

### ステップ 2: スプレッドシートから企業リスト取得

1. `browser_navigate` で `config.spreadsheet.url` に遷移する
2. `browser_snapshot` でスプレッドシートの構造を確認する
3. `config.spreadsheet.columns` の定義に従い、A列（企業名）・B列（送信済みチェック）・C列（フォームURL）を読み取る
4. B列が未チェック（空欄またはFALSE）の行を全件抽出し、企業リストを作成する
5. `--company` オプション指定時は該当企業名の行のみを対象とする

### ステップ 3: 企業ループ開始

未送信企業を先頭から順番に処理する。バッチ処理の効率を最大化するため、個々のフォーム入力・送信にはユーザー確認を挟まない（未知フィールドへの質問のみが中断ポイント）。

進捗は都度表示してユーザーが状況を把握できるようにする：

```
[3/150] 株式会社XXX を処理中...
```

成功した企業名はメモリ内リストに追記し、ループ終了後にスプレッドシートをまとめて更新する（1社ごとにスプレッドシートへ戻るとブラウザ往復コストが高いため）。

## 1社あたりのステップ

### ステップ 4: フォームURLに遷移

`browser_navigate` でC列のURLに遷移する。

### ステップ 5: フォーム検出

1. `browser_snapshot` でページ構造を取得する
2. `<input>` / `<textarea>` / `<select>` 等のフォームフィールドが存在するか確認する
3. フィールドが見つかった場合: そのままステップ6へ進む
4. フィールドが見つからない場合: フォーム探索ロジック（後述）を実行する

### ステップ 6: フィールド照合

`browser_snapshot` で取得したフィールド一覧と `commonData` のキーを意味的にマッチングする。
Claude 自身がフォームのラベル名と `commonData` キーの意味的対応を判断する。

意味的マッチングの例：
- 「御社名」「貴社名」→ `commonData["会社名"]`
- 「お名前」「担当者名」→ `commonData["氏名"]`
- 「E-mail」「メール」→ `commonData["メールアドレス"]`
- 「TEL」「電話」→ `commonData["電話番号"]`
- 「ご用件」「お問い合わせ種別」→ `commonData["お問い合わせ種別"]`

### ステップ 7: 未知フィールド対応

各フィールドについて以下を判断する：

**マッチ済みフィールド:** `commonData` の対応値を自動使用する

**未知フィールド（`commonData` にも意味的に近いキーがない場合）:**
1. `field-log.json` の `entries` を検索し、同じまたは類似した `fieldLabel` のエントリを探す
2. ログに記録あり → そのエントリの `value` を使用し、`count` を +1、`companies` に当該企業名を追加する
3. ログに記録なし → `AskUserQuestion` ツールでユーザーに質問する：

```
[未知フィールド] 「{フィールドラベル}」への入力値を教えてください。
（このフォームでは選択肢: {選択肢リスト} があります）
```

ユーザーの回答を `field-log.json` に新規エントリとして追加する（`count: 1`）。

### ステップ 8: フォーム入力

フィールドの種類に応じて入力する：
- テキスト入力欄: `browser_fill_form` または `browser_type`
- チェックボックス・ラジオボタン: `browser_click`
- セレクトボックス: `browser_select_option`

バッチ処理の効率を最大化するため、個々の入力にユーザー確認は挟まない。

### ステップ 9: 送信・確認画面対応

日本のフォームは「確認画面→送信」の2段階が一般的なため、完了まで最大3回のクリックを試みる：

1. 送信ボタン（「送信」「Submit」「確認する」「次へ」等）を `browser_click` でクリック
2. `browser_snapshot` でページ状態を確認する
3. **確認画面の場合**（「入力内容の確認」「以下の内容で送信します」等が表示）:
   「送信する」「上記内容で送信」「確定する」等のボタンを `browser_click` でクリック
4. `browser_snapshot` を再取得し、完了画面になるまで繰り返す（最大3回）

### ステップ 10: 結果確認

1. `config.options.screenshotAfterSubmit` が `true` の場合は `browser_take_screenshot` でスクリーンショットを撮影する
2. `browser_snapshot` で送信後のページ状態を確認する
3. 以下のいずれかで成功を判断する：
   - 「送信完了」「ありがとうございました」「お問い合わせを受け付けました」等のメッセージが表示されている
   - URLが完了ページ（`/thanks`、`/complete`、`/done` 等）に遷移している
4. 成功と判断できない場合はエラーとして処理する

### ステップ 11: エラー時の処理

`config.options.skipOnError: true`（デフォルト）の場合:
- エラー内容（企業名・エラー種別・詳細）をメモリ内のエラーリストに追加する
- スプレッドシートのB列は更新しない
- 次の企業の処理に進む

`config.options.skipOnError: false` の場合:
- エラーメッセージを表示してバッチ処理を停止する

---

## ループ終了後の処理

### ステップ 12: スプレッドシートを一括更新

個々の送信後に毎回スプレッドシートへ戻るとブラウザ往復コストが高くなるため、全社処理完了後にまとめてB列を更新する：

1. `browser_navigate` で `config.spreadsheet.url` に遷移する
2. `browser_snapshot` でスプレッドシートの状態を確認する
3. メモリ内の成功企業リストを順番に処理し、A列を検索して各企業の行を特定する
4. B列のチェックボックスセルを `browser_click` でクリックしてONにする
5. 全件完了後に `browser_snapshot` で変更が反映されたことを確認する

### ステップ 13: フィールドログ保存

処理中に更新した `field-log.json` の内容を Write ツールで保存する。
パス: `~/.claude/form-submit/field-log.json`

### ステップ 14: config昇格提案

`field-log.json` の `entries` の中で `count >= 3` のエントリを抽出し、以下の形式でユーザーに提案する：

```
以下のフィールドが3回以上出現しています。config.json の commonData に追加しますか？

- 「部署名」→ 値: 営業部（5回出現）
- 「ご担当者名カナ」→ 値: ヤマダ タロウ（3回出現）

[Y] すべて追加  [n] スキップ  または追加するフィールドを番号で指定してください
```

ユーザーが承認した場合は `config.json` の `commonData` に該当キーバリューを追加して Write ツールで保存する。

### ステップ 15: 最終レポート表示

```
=== 送信結果 ===
成功: 18社
スキップ（フォーム未検出）: 3社
エラー: 2社
---
エラー詳細:
- 株式会社XXX: CAPTCHA検出
- 株式会社YYY: タイムアウト
---
config昇格候補（3回以上出現）:
- 「部署名」→ 営業部（5回出現）→ config.jsonに追加しますか？
```

## フォーム探索ロジック

URLがフォームページでない場合（ステップ5でフォームが見つからない場合）に実行する。

1. `browser_snapshot` の現在のページから以下のキーワードを含むリンクを検索する：
   「お問い合わせ」「Contact」「パートナー」「協業」「提携」「SES」「採用」「ビジネス」
2. 最も関連性が高いリンクを選択し、`browser_click` で遷移する
3. 遷移先で `browser_snapshot` を再取得し、フォームフィールドの有無を確認する
4. フォームが見つかればステップ6へ進む
5. 1階層目でも見つからない場合: 同様のキーワードで2階層目まで探索する
6. 2階層探索してもフォームが見つからない場合: スキップしてエラーリストに「フォーム未検出」として記録する

## エラーハンドリング

### CAPTCHA検出

`browser_snapshot` の結果に reCAPTCHA の iframe / hCaptcha の要素 /「ロボットではありません」チェックボックスが含まれる場合:
→ スキップしてエラーリストに「CAPTCHA検出」として記録する

### タイムアウト

`browser_navigate` または `browser_snapshot` が想定時間内に完了しない場合:
→ スキップしてエラーリストに「タイムアウト」として記録する

### ログイン要求

`browser_snapshot` の結果にログインフォーム（`type="password"` フィールドや「ログイン」「サインイン」ボタン）が含まれる場合:
→ スキップしてエラーリストに「ログイン要求」として記録する

### 送信エラー

送信後のページに「エラー」「入力内容をご確認」「必須項目」「送信に失敗」等のメッセージが含まれる場合、またはフォームが再表示されている場合:
→ エラーメッセージの内容をログに記録してスキップする

### フォーム未検出

2階層探索してもフォームが見つからない場合:
→ スキップしてエラーリストに「フォーム未検出」として記録する

---

## 注意事項

- スプレッドシートの操作は Google Sheets API ではなく、Playwright MCP 経由のブラウザ操作で行う
- 未知フィールドへの質問には `AskUserQuestion` ツールを使用する
- 処理中は成功企業リストをメモリに保持し、ループ終了後にスプレッドシートへ一括反映する（毎回戻ると非効率）
